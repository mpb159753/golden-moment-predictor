# GMP-Daily 文案生成增强：上下文注入、数据优化与历史归档

## 背景

当前 `gmp-daily` 工作流每天将 `summary_3day.json` 数据转化为小红书三日攻略文案。存在以下改进方向：

1. **缺少时间上下文**：文案不知道当天是周几、是否是节假日，导致无法做差异化表达
2. **无历史文案参考**：连续几天的文案可能用词重复、结构雷同，且生成后未归档保存
3. **数据结构不匹配消费场景**：`summary_3day.json` 的扁平列表结构与文案"按日分层"的需求不一致，存在冗余字段和低价值数据

## 设计目标

- 优化 `summary_3day.json` 的数据结构，使其直接服务于文案生成
- 在 `buildSummary()` 中集成时间上下文计算（周几、法定节假日、节气、旅游旺淡季）
- 建立历史文案归档机制，支持手动确认后保存
- 生成时自动读取最近 3 天历史文案，注入 Prompt 避免内容重复

## 模块一：`summary_3day.json` 数据结构优化

对 `frontend/scripts/build-summary.mjs` 中的 `buildSummary()` 函数进行改造。

### 1.1 按日期分组 + 预分层（P0）

**问题**：当前 `推荐点位` 是按分数降序的扁平列表（120+ 条），但文案 Prompt 要求"按日期为主线、逐日分板块呈现"，LLM 每次都需要二次加工。

**改造**：输出改为按日期分组 + 按分层规则预分类：

```json
{
  "每日详情": {
    "2026-03-02": {
      "必冲": [
        { "区域": "其它景区", "点位": "红杉林主机位", "景观": "树挂积雪", "天气": "阴天", "时段": "上午", "分数": 88 }
      ],
      "择优": [ ... ],
      "观望": [ ... ]
    },
    "2026-03-03": { ... },
    "2026-03-04": { ... }
  }
}
```

分层规则：
- **必冲**：score ≥ 85
- **择优**：score 60-84，晴天/少云 + 高优先级景观
- **观望**：score 60-84 中不满足择优条件的，或虽满足但分数较低

### 1.2 集成日期上下文（P0）

**改造**：在 `buildSummary()` 中计算时间上下文，嵌入 JSON 输出。查表数据从 `config/` 目录的配置文件读取。

```json
{
  "日期上下文": [
    { "date": "2026-03-02", "weekday": "周一", "holiday": null, "solar_term": null, "season": "淡季" },
    { "date": "2026-03-03", "weekday": "周二", "holiday": null, "solar_term": null, "season": "淡季" },
    { "date": "2026-03-04", "weekday": "周三", "holiday": null, "solar_term": "惊蛰", "season": "淡季" }
  ]
}
```

上下文信息维度：

| 维度 | 数据来源 | 说明 |
|------|----------|------|
| 星期几 | `new Date().getDay()` 计算 | 100% 准确 |
| 法定节假日 | `config/chinese_holidays.json` | 按年维护，含调休标记 |
| 节气 | `config/solar_terms.json` | 24 节气日期，每年固定 |
| 旺淡季 | `config/tourism_seasons.json` | 旺/平/淡季时段定义 |

### 1.3 删除 `各区概况` 冗余字段（P1）

**问题**：`各区概况` 只记录每个区域的单个最高分点位，无日期维度，与 `推荐点位` 完全冗余，文案 Prompt 中没有直接使用规则。

**改造**：删除此字段。

### 1.4 过滤低价值条目（P1）

**问题**：当前保留所有 ≥ 60 分的条目（120+ 条），但大量 60-65 分 + 阴天/雨天的条目在文案中不会被推荐，白白消耗 Token。

**改造**：在预分层时，`观望` 层级仅保留 top N 条（如前 5 条），其余丢弃。必冲和择优层级保持全量。

### 1.5 特定景观增加推荐时间提示（P2）

**改造**：对日照金山、观星等时间敏感景观，增加 `推荐时间` 字段：

```json
{ "景观": "日照金山", "推荐时间": "日出前30分钟到位", "时段": "上午" }
```

### 1.6 改造后的完整输出结构

```json
{
  "生成时间": "2026-03-02T20:00:00+08:00",
  "适用日期": "3月3日—3月5日",
  "日期上下文": [
    { "date": "2026-03-03", "weekday": "周一", "holiday": null, "solar_term": null, "season": "淡季" },
    { "date": "2026-03-04", "weekday": "周二", "holiday": null, "solar_term": null, "season": "淡季" },
    { "date": "2026-03-05", "weekday": "周三", "holiday": null, "solar_term": "惊蛰", "season": "淡季" }
  ],
  "每日详情": {
    "2026-03-03": {
      "必冲": [
        { "区域": "其它景区", "点位": "红杉林主机位", "景观": "冰挂", "天气": "雪", "时段": "上午", "分数": 89 }
      ],
      "择优": [
        { "区域": "亚丁景区", "点位": "五色海", "景观": "观星", "天气": "晴天", "时段": "下午", "分数": 74 }
```

## 模块二：历史文案归档

### 2.1 存储位置

项目根目录下 `daily_posts/` 目录，每天一个 Markdown 文件：

```
daily_posts/
├── 2026-03-01.md
├── 2026-03-02.md
└── ...
```

### 2.2 文件格式

```markdown
---
date: 2026-03-02
covers: [2026-03-02, 2026-03-03, 2026-03-04]
weekday: 周一
holiday: null
solar_term: null
season: 淡季
---

# 标题

（文案正文）
```

- YAML frontmatter 记录生成时的 context 元数据
- `covers` 字段标记文案覆盖的日期范围，便于回溯时识别内容重叠
- 正文即最终确认发布的版本

### 2.3 归档触发方式

手动确认后归档：
- 工作流生成文案后，提示用户确认
- 用户满意后说"归档"，工作流将最终版本写入 `daily_posts/YYYY-MM-DD.md`
- 归档的是用户最终确认的版本，不是 AI 首次生成的原始输出

## 模块三：历史文案注入

### 3.1 回溯策略

- 固定回溯最近 **3 天**
- 有几篇读几篇（不足 3 篇也正常运行）
- 3 天窗口刚好覆盖三日文案的日期重叠周期

### 3.2 注入 Prompt 方式

在核心 Prompt 模板中新增一个段落：

```markdown
## 【近期已发布文案参考（仅用于避免重复，禁止抄袭）】
以下是最近发布的文案，请注意：
1. 避免与以下文案使用相同的情绪钩子和形容词
2. 对于日期重叠的点位，换一种角度描述
3. 保持整体风格一致但表达求新

[最近3天文案内容]
```

### 3.3 Token 消耗估算

- 每篇文案正文 ≤ 800 字符
- 3 篇历史文案 ≈ 2400 字符
- 加上 context 信息 ≈ 3000 tokens
- 对整体 Prompt 长度影响可控

## 工作流变更总结

增强后的 `gmp-daily` 工作流步骤：

```
① 读取 summary_3day.json（已有，数据结构已优化）
   • 数据已按日期分组 + 预分层（必冲/择优/观望）
   • 已包含日期上下文（周几/节假日/节气/旺淡季）
     ↓
② 读取最近 3 天历史文案（新增）
   • 从 daily_posts/ 读取
     ↓
③ 组合 Prompt 生成文案（增强）
   • 在已有 Prompt 中注入历史文案参考段落
   • 日期上下文直接从 JSON 获取，无需额外计算
     ↓
④ 输出文案，用户手动确认后归档（新增）
   • 保存到 daily_posts/YYYY-MM-DD.md
```
